## 4. Issues

### Backend Issues

**BE-5 — Medium: `start_time` window too tight for slow signers**  
`src/orchestrator/intent.ts:231, 287` — `start_time = now + 30s`. If the user takes longer than 30 seconds to sign (hardware wallet, slow network), the order's `start_time` is already in the past at submission. The Cairo contract may reject it. Consider using `now - 60` (allow orders to be valid from 60 seconds ago) or compute `start_time` at submission time rather than intent creation time.

**BE-6 — Low: `tokenStandard` not enum-validated on offer/fulfill endpoints**  
`src/api/routes/intents.ts:47–50` — `tokenStandard: z.string().optional()`. A typo like `"ERC-1155"` silently routes to ERC-721 handling. Fix: `.enum(["ERC721", "ERC1155"])`.

**BE-7 — Medium: Missing `approve` pre-call when order not in DB and no `tokenStandard` hint**  
`src/orchestrator/intent.ts:393` — `buildFulfillOrderIntent` falls back to ERC-721 handling when the DB lookup returns null. If `order` is null and no `tokenStandard` hint is provided, neither `isListing` nor `isOffer` is true, so no `approve` call is generated. The client must always send `tokenStandard` for orders not yet indexed.

**BE-8 — Low: Same enum-validation gap on fulfill endpoint**  
Same pattern as BE-6, applies to `fulfillSchema`.

**BE-9 — Medium: `ORDER_STATUS_INDEX.erc1155 = 14` not verified against Medialane1155V2 ABI**  
`src/utils/txVerifier.ts:257` — The stale-cancel recovery reads `get_order_details` response at index 14 for `order_status` — the same index as ERC-721. If `Medialane1155V2`'s `get_order_details` has a different response layout, stale-cancel recovery for ERC-1155 silently fails. **Action:** verify against the deployed contract ABI.

**BE-10 — Medium: ERC-1155 `OrderFulfilled` event parsed from raw data without validation**  
`src/mirror/handlers/orderFulfilled1155.ts:34` — `rawEvent.data[1]` assumed to be `remaining_amount`. No length/type check. If the event layout changes, silent misinterpretation occurs.

**BE-11 — Low: `fetchOrderDetails1155` does not parse indices 14–15**  
`src/mirror/handlers/orderCreated1155.ts` — The 17-felt response has unparsed fields at positions 14–15. Currently harmless (resync uses stored DB fields), but may need updating if those fields are needed.

**BE-1 through BE-4** — Low/cosmetic: `SNIP12_TYPES_1155` is a direct reference to `SNIP12_TYPES` (not a copy — cosmetic only); no ERC-1155 collection creation endpoint (ERC-1155 collections bypass the intent API — confirmed by design); no ERC-1155 mint intent endpoint (same — by design); no RPC timeout on `buildMintIntent`'s `is_collection_owner` call.

### SDK Issues

**SDK-1 — Low: `cancelPayload` key spelled `cancelation` (single 'l')**  
`src/marketplace1155/orders.ts` — `cancelPayload = { cancelation: cancelParams, signature }`. The SDK direct-execution flow (non-ChipiPay) uses `contract.populate("cancel_order", [cancelPayload])`. If the Cairo ABI entry point expects `cancellation` (double 'l'), the call will fail silently. **Action:** verify the Medialane1155V2 ABI parameter name for `cancel_order`. This only affects the SDK's direct `cancelOrder1155()` function — ChipiPay flows go through the backend intent system which has its own calldata layout.

### Frontend Issues (medialane-io)

**IO-1 — Medium: `COLLECTION_1155_CONTRACT` not passed to `MedialaneClient` constructor**  
`src/lib/medialane-client.ts:15–25` — `collection1155Contract` is never passed. The ERC-1155 collection creation page (`/launchpad/nfteditions/create`) reads `COLLECTION_1155_CONTRACT` directly from constants (bypassing the SDK client), so it is currently unaffected. But if any SDK method ever needs this address, it will use the SDK's baked-in default rather than the env var.  
**Fix:** Add `collection1155Contract: COLLECTION_1155_CONTRACT` to the `MedialaneClient` constructor call.

**IO-2 — Medium: No backend sync after ERC-1155 collection deploy**  
`src/app/launchpad/nfteditions/create/page.tsx` (~line 160) — After `executeTransaction` succeeds, the page shows success but never calls `syncCollectionFromTx(txHash)`. The new collection will not appear in `useCollectionsByOwner` until the next indexer cycle. Compare: ERC-721 collection creation calls `syncCollectionFromTx` immediately.  
**Fix:** Call `await syncCollectionFromTx(txHash)` (or a 1155-equivalent) after the deploy tx is confirmed.

**IO-3 — Low: `amount` silently dropped in `createListing` when falsy for ERC-1155**  
`src/hooks/use-marketplace.ts:392` — `...(is1155 && input.amount ? { amount: input.amount } : {})`. The listing dialog prevents this with form validation, but calling the hook directly without `amount` sends an incomplete request. **Fix:** Default to `"1"` in the hook: `amount: is1155 ? (input.amount || "1") : undefined`.

**IO-4 — Low–Medium: Offer dialog has no quantity selector for ERC-1155**  
`src/components/marketplace/offer-dialog.tsx` + `src/hooks/use-marketplace.ts:476` — The offer dialog does not show an amount input for ERC-1155 tokens, and the hook hardcodes `quantity: "1"`. Users cannot make multi-unit offers on editions. The listing dialog correctly shows a quantity input for ERC-1155. **Fix:** Add optional `amount` field to `MakeOfferInput` and show a quantity input in the offer dialog when `resolvedStandard === "ERC1155"`.

**IO-5 — Low: ERC-1155 mint has no backend confirmation poll and no portfolio cache invalidation**  
`src/app/launchpad/nfteditions/[contract]/mint/page.tsx:162–167` — After ChipiPay confirms the tx, the page shows success without polling the backend for indexer confirmation or calling `invalidatePortfolioCache`. The new token may not appear in portfolio immediately.

**IO-6 — Low: `usePostMintListing` drops `tokenStandard`**  
`src/hooks/use-post-mint-listing.ts:71–78` — `createListing()` called without `tokenStandard`, defaulting to ERC-721. Safe as long as this hook is only used from the ERC-721 create page. Would silently route ERC-1155 post-mint listings to the wrong contract if reused. **Fix:** Accept and forward a `tokenStandard` prop.

**IO-7 — Very Low: List button enabled when standard is `"UNKNOWN"`**  
`src/hooks/use-resolved-token-standard.ts:15–17` + `src/components/marketplace/listing-dialog.tsx:106` — `standardResolved` is `true` when `resolvedStandard === "UNKNOWN"` (non-null), so the listing button is not disabled for tokens whose standard has not been indexed yet. The listing attempt would reach the backend with an ambiguous standard.
